# Ship to Production

Complete shipping workflow that handles merging feature branches to develop and creating release PRs to main.

**Arguments:** `$ARGUMENTS` (optional: `help`, target branch like `main` or `develop`, or release title)

## Workflow Overview

The ship skill handles two scenarios:

### Scenario A: From a Feature Branch (Worktree)
1. Push the feature branch to origin
2. Create PR targeting `develop`
3. Auto-merge the PR to develop
4. Ask if user wants to continue shipping develop тЖТ main
5. If yes, create release PR with changelog

### Scenario B: From develop or main branch
1. Create release PR from `develop` тЖТ `main` with changelog

## Steps to Execute

### Step 1: Detect Current Context

```bash
git branch --show-current
```

Determine if we're on:
- A **feature branch** (e.g., `feature/*`, `fix/*`, `bugfix/*`) тЖТ Scenario A
- The **develop** branch тЖТ Scenario B
- The **main** branch тЖТ Nothing to ship, inform user

Also check if we're in a worktree:
```bash
git rev-parse --git-common-dir
```

### Step 2: Fetch Latest

```bash
git fetch origin
```

---

## Scenario A: Feature Branch Workflow

### Step A1: Check for Uncommitted Changes

```bash
git status --porcelain
```

If there are uncommitted changes, warn the user and ask if they want to continue (changes won't be included).

### Step A2: Push Feature Branch

```bash
git push -u origin <branch-name>
```

### Step A3: Create PR to Develop

Check if a PR already exists:
```bash
gh pr list --head <branch-name> --base develop --json number,url
```

If no PR exists, create one:
```bash
gh pr create \
  --base develop \
  --head <branch-name> \
  --title "<PR title from branch name or last commit>" \
  --body "## Summary
<auto-generated from commits on this branch>

## Test plan
- [ ] Tested locally
- [ ] Ready for review"
```

### Step A4: Auto-Merge to Develop

Merge the PR using squash merge:
```bash
gh pr merge <pr-number> --squash --delete-branch
```

If merge fails (conflicts, CI failing), inform the user and stop. They'll need to resolve manually.

### Step A4.5: Ensure Branch Deletion

The `--delete-branch` flag should delete the remote branch, but if it fails (e.g., due to local checkout issues), explicitly delete it:

```bash
git push origin --delete <branch-name>
```

This ensures the branch is cleaned up even without GitHub auto-delete configured.

### Step A5: Ask About Release PR

After successful merge to develop, ask the user:

> "тЬЕ Merged to develop! Would you like to create a release PR from develop тЖТ main?"

**Options:**
1. **Yes, create release PR** тЖТ Continue to Scenario B
2. **No, just the develop merge** тЖТ Stop here and report success

If `$ARGUMENTS` is `main`, skip the question and proceed to release PR.
If `$ARGUMENTS` is `develop`, skip the question and stop after develop merge.

## Scenario B: Release PR Workflow (develop тЖТ main)

### Step B1: Check develop is ahead of main

```bash
git rev-list --count origin/main..origin/develop
```

If the count is 0, inform the user there's nothing to ship.

### Step B2: Get Commits for Changelog

```bash
git log origin/main..origin/develop --pretty=format:"%h|%s|%an" --reverse
```

### Step B3: Generate Changelog

Parse commits and group them by type based on commit message patterns:

**Categories to detect:**
- ЁЯЪА **Features** - commits starting with: `Add`, `Implement`, `Create`, `feat:`
- ЁЯРЫ **Bug Fixes** - commits starting with: `Fix`, `fix:`, `Bugfix`, `Resolve`
- тЩ╗я╕П **Refactoring** - commits starting with: `Refactor`, `refactor:`, `Clean`, `Simplify`
- ЁЯУж **Dependencies** - commits mentioning: `dependency`, `dependencies`, `package`, `upgrade`, `npm`, `composer`
- ЁЯФз **Configuration** - commits starting with: `Configure`, `config:`, `Update config`, `chore:`
- ЁЯУЭ **Documentation** - commits starting with: `docs:`, `Doc`, `Update README`, `Update CLAUDE`
- ЁЯОи **Styling** - commits starting with: `style:`, `Style`, `CSS`, `Tailwind`, `UI`
- тЬЕ **Tests** - commits starting with: `test:`, `Test`, `Add test`
- ЁЯЧГя╕П **Database** - commits mentioning: `migration`, `database`, `schema`
- ЁЯФе **Removals** - commits starting with: `Remove`, `Delete`, `Drop`
- ЁЯУж **Other** - anything that doesn't match the above

### Step B4: Determine Release Title

If `$ARGUMENTS` was provided (and isn't `main` or `develop`), use that as the release title.

Otherwise, suggest a title based on the most significant changes and ask the user to confirm or customize.

### Step B5: Create Release PR

```bash
gh pr create \
  --base main \
  --head develop \
  --title "Release: <title>" \
  --body "<changelog>"
```

**PR Body Format:**

```markdown
## ЁЯЪА Release: <title>

<one-line summary of what's shipping>

---

## Changelog

### ЁЯЪА Features
- <commit message> (<hash>)
- ...

### ЁЯРЫ Bug Fixes
- <commit message> (<hash>)
- ...

### тЩ╗я╕П Refactoring
- ...

(only include sections that have commits)

---

## Stats

- **Commits:** X
- **Contributors:** <list>
- **Files Changed:** X

---

## Deployment Checklist

- [ ] All tests passing
- [ ] Reviewed changelog for accuracy
- [ ] Ready to merge
```

### Step B6: Report Success (NEVER AUTO-MERGE)

**тЪая╕П CRITICAL: DO NOT merge the release PR automatically.**

The release PR to main must ALWAYS be reviewed and merged manually by the developer. Even if the user says "ship to main" or "merge it", you should:

1. Create the PR only
2. Provide the PR URL
3. Remind the user to review and merge when ready
4. **STOP** - do not run `gh pr merge` for PRs targeting main

Provide the user with:
- PR URL
- Summary of what's being shipped
- Clear reminder: "Review the PR and merge when ready - I won't auto-merge to main"

## Important Notes

- **Auto-merge to develop**: The PR to develop is automatically merged (squash) if CI passes
- **тЪая╕П NEVER AUTO-MERGE TO MAIN**: The develop тЖТ main PR is created but MUST NOT be auto-merged. Always stop after creating the PR and let the developer review and merge manually. This is a safety requirement.
- **Changelog is editable**: The auto-generated changelog can be edited in the PR before merging
- **Conflicts**: If merge conflicts occur, the skill will stop and you'll need to resolve manually
- **CI failures**: If CI fails on the develop PR, the merge will fail and you'll be notified
